Publicación

Autores

Princeton University &
Google Research

Referencia

Contexto
GPT-3 y PaLM ya existían. El reto: hacerlos más fiables y conectados al mundo real.
¿Qué problema venía a resolver?
🧠
PROBLEMA 1 · CHAIN-OF-THOUGHT
Razonaba, pero no podía actuar
CoT permitía al modelo "pensar paso a paso", pero todo surgía de sus parámetros. Sin acceso a datos externos, el modelo inventaba hechos cuando no los conocía — las famosas alucinaciones.
⚙️
PROBLEMA 2 · MODELOS SOLO-ACCIÓN
Actuaba, pero sin razonar
Los pipelines de herramientas ejecutaban acciones, pero el modelo no explicaba por qué las elegía. Sin trazabilidad, era imposible depurar errores o recuperarse si una acción fallaba.
✦
La propuesta de ReAct
Entrelazar razonamiento y acción en un único bucle: el modelo razona, actúa, observa el resultado real y razona de nuevo — anclando cada paso a datos verificables.

¿Qué es ReAct?

RE
ACT
REASONING
ACTING
Yao et al. · Princeton / Google Research · 2022
arXiv:2210.03629
💡
CONCEPTO

Pensar en voz alta

Es un método que permite a la IA "pensar en voz alta" antes de ejecutar una tarea, haciendo visible su razonamiento paso a paso.

⚡
EL DIFERENCIAL

Reasoning + Acting entrelazados

A diferencia de un prompt común, ReAct obliga al modelo a generar una traza de razonamiento seguida de una acción específica — buscar en internet, usar una calculadora, llamar una API.

🎯
BENEFICIO

Menos alucinaciones

Reduce errores porque cada afirmación se ancla a una observación real. Permite resolver problemas que requieren datos externos que el modelo no tiene en sus parámetros.

Estructura del Prompt ReAct

Cada respuesta ReAct sigue esta traza — el modelo la genera de forma iterativa hasta tener suficiente información:
Pregunta: El problema o tarea que el usuario quiere resolver.
Thought: La IA razona en voz alta — explica qué necesita y por qué, antes de hacer nada.
Action: Elige una herramienta disponible (search_web, read_url, calculator…)
Action Input: El parámetro concreto que se le pasa a la herramienta elegida.
Observation: El resultado real devuelto por la herramienta. Se incorpora al contexto del modelo.
Final Answer: La conclusión basada en todas las observaciones. Verificada con datos reales.
↺   Los pasos Thought → Action → Observation se repiten hasta tener información suficiente
Componentes del ciclo
Pregunta
Entrada del usuario. Define el objetivo completo del agente.
Thought
Razonamiento interno. El modelo planifica su siguiente paso.
Action
Herramienta seleccionada para obtener información externa.
Action Input
Argumento concreto enviado a la herramienta.
Observation
Resultado real devuelto. Alimenta el siguiente Thought.
Final Answer
Respuesta definitiva, basada en observaciones verificadas.
¿Cuántas iteraciones?
No hay límite fijo. El modelo decide cuándo tiene suficiente información para emitir la Final Answer. Cada iteración añade contexto verificado.

✦ Abrir Claude
# ── INSTRUCCIÓN DE SISTEMA ──────────────────────────────── Eres un agente ReAct con acceso a herramientas web y de documentos. SIEMPRE responde siguiendo este formato exacto, sin saltarte pasos: ────────────────────────────────────────────────────────── Thought: <razona en voz alta qué necesitas buscar y por qué> Action: search_web | read_url | create_document Action Input: <parámetro concreto para la herramienta> Observation: <resultado devuelto por la herramienta — lo proporciona el sistema> ... (repite Thought → Action → Observation cuantas veces necesites) Thought: Ya tengo toda la información. Voy a crear el documento. Final Answer: <confirmación de que el PDF fue creado y resumen de lo que contiene> ────────────────────────────────────────────────────────── # ── HERRAMIENTAS DISPONIBLES ────────────────────────────── search_web(query: str) → busca información actualizada en internet read_url(url: str) → lee y extrae el contenido de una página web create_document(titulo, contenido) → genera un PDF estructurado con el resultado # ── REGLAS ──────────────────────────────────────────────── 1. Nunca inventes datos — si no sabes algo, usa search_web primero. 2. Incluye la URL fuente de cada dato que uses en el documento. 3. No emitas Final Answer sin antes ejecutar create_document. 4. El PDF debe tener: portada, cronograma clasificado y fuentes. ────────────────────────────────────────────────────────── # ── TAREA ───────────────────────────────────────────────── El Mundial FIFA 2026 acaba de comenzar. Quiero que: 1. Busques el calendario completo de partidos del Mundial 2026 (fechas, sedes, grupos y equipos confirmados) 2. Clasifiques cada partido en una de estas 3 categorías: 🔥 IMPERDIBLE — clásicos, rivalidades, favoritos al título ⚡ PUEDE SORPRENDER — equipos que podrían dar la campanada 😴 MENOS ATRACTIVO — menor interés competitivo o mediático 3. Generes un PDF titulado "Guía del Mundial FIFA 2026" con: - Cronograma clasificado por categoría - Fecha, hora y sede de cada partido destacado - Breve análisis (1-2 líneas) de por qué es o no interesante - Sección especial: "Los 5 partidos que no te puedes perder" - Fuentes consultadas al final del documento
Qué es cada parte
Instrucción sistema
Le dice al modelo su rol y el formato obligatorio que debe seguir.
Thought:
El modelo razona en voz alta antes de actuar. Planificación explícita.
Action:
La herramienta que elige usar para obtener info externa.
Action Input:
El argumento exacto pasado a la herramienta (query, URL…)
Observation:
Resultado real de la herramienta. Alimenta el siguiente razonamiento.
Final Answer:
Respuesta definitiva emitida solo después de verificar con datos reales.
Tarea del usuario
Lo único que cambia entre usos. Define qué debe investigar y generar.
